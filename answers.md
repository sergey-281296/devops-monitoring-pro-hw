# Ответы на вопросы по системам мониторинга

## 1. Минимальный набор метрик для платформы вычислений

**Необходимые метрики:**
- **CPU usage (загрузка ЦПУ)** - так как вычисления загружают процессор, это критический показатель
- **RAM usage (использование памяти)** - контроль потребления оперативной памяти
- **Disk space (свободное место на диске)** - отчеты сохраняются на диск, нужно следить за заполнением
- **HTTP response codes (коды ответов HTTP)** - мониторинг доступности сервиса (2xx, 4xx, 5xx)
- **HTTP response time (время ответа)** - производительность сервиса
- **System load average** - общая нагрузка на систему

**Почему именно эти:** Они показывают здоровье системы, производительность вычислений и доступность сервиса для пользователей.

---

## 2. Бизнес-метрики для менеджера продукта

**Предложить метрики:**
- **SLA (Service Level Agreement)** - процент успешных запросов (например, 99% времени сервис доступен)
- **Average report generation time** - среднее время генерации отчета
- **Requests per hour/day** - количество обработанных запросов
- **Service Availability** - процент времени работы без сбоев
- **Error rate** - процент ошибок от общего числа запросов
- **User satisfaction metrics** - если есть возможность сбора обратной связи

**Объяснение для менеджера:** Вместо технических терминов (RAM, inodes, CPUla) показываем понятные бизнес-показатели: "Система работает 99% времени", "Отчет генерируется в среднем за 3 секунды", "Сегодня обработано 5000 запросов".

## 3. Сбор логов без финансирования

**Решения:**
- Использовать **стандартный вывод приложений** (stdout/stderr) и перенаправлять в файлы
- Настроить **logrotate** для ротации логов и экономии места
- Развернуть бесплатные open-source решения:
  - **ELK Stack** (Elasticsearch + Logstash + Kibana)
  - **Grafana Loki** - легковесная система для логов
  - **Graylog**
- Настроить **уведомления об ошибках** через email, Slack или Telegram
- Использовать **application-level logging** с отправкой критических ошибок разработчикам

**Преимущество:** Все решения бесплатные и open-source.

## 4. Ошибка в формуле SLA

**Проблема:** Формула `summ_2xx_requests / summ_all_requests` считает только 2xx коды успешными.

**Ошибка:** Не учитываются **3xx коды** (редиректы), которые также являются корректными ответами сервера.

**Правильная формула:** 
SLA = (summ_2xx + summ_3xx) / summ_all_requests * 100%

**Почему 70%:** Вероятно, 30% запросов возвращают 3xx коды (редиректы), которые система ошибочно считает неуспешными. Также возможно, что health check запросы от мониторинга искажают статистику.

## 5. Pull vs Push системы мониторинга

### Pull-системы (Prometheus, VictoriaMetrics)
**Плюсы:**
- Центральный контроль сбора данных
- Легче масштабировать и управлять конфигурацией
- Безопаснее - не нужно открывать входящие порты
- Проще отладка и мониторинг самих collectors

**Минусы:**
- Нагрузка на сервер при большом количестве targets
- Сложнее с динамическими средами (контейнеры)
- Задержки между интервалами скрапинга

### Push-системы (Telegraf → InfluxDB, StatsD)
**Плюсы:**
- Агенты сами отправляют данные
- Меньше нагрузка на центральный сервер
- Лучше для динамических и масштабируемых сред
- Данные отправляются сразу при событии

**Минусы:**
- Нужно открывать порты для приема данных
- Риск потери данных при недоступности сервера
- Сложнее контролировать и валидировать входящий поток

## 6. Классификация систем

| Система | Модель | Описание |
|---------|--------|----------|
| **Prometheus** | Pull | Сервер сам запрашивает метрики у targets |
| **TICK Stack** | Push | Telegraf отправляет данные в InfluxDB |
| **Zabbix** | Гибридная | Поддерживает оба режима (active/passive checks) |
| **VictoriaMetrics** | Pull | Совместима с Prometheus, pull-модель |
| **Nagios** | Pull | Активно проверяет сервисы по расписанию |

**Гибридные системы:** Zabbix может работать как в pull, так и в push режиме.
